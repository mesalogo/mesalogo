## KISS原则违背点

### ⚠️ 违背1: 过多配置参数
**问题：**
- BM25 k1/b、RRF k 这些参数需要专业知识
- 95%用户不知道这些参数的含义
- 调参需要反复测试，成本高

**改进建议：**
```diff
- 暴露 BM25 k1, b 参数给用户
+ 使用学术界推荐的默认值 (k1=1.5, b=0.75)
+ 只在代码中可修改，不在UI中暴露
```

### ⚠️ 违背2: 两种融合策略
**问题：**
- 用户不理解"加权融合"和"RRF融合"的区别
- 两种算法都能工作，提供选择反而增加决策成本
- 根据搜索引擎最佳实践，加权融合更直观

**改进建议：**
```diff
- 提供两种融合策略供用户选择
+ 只使用加权融合 (weighted fusion)
+ 默认向量权重70%，允许用户调整
+ RRF作为未来可选项，当前不暴露
```

### ⚠️ 违背3: 前端UI过于复杂
**问题：**
- 嵌套的动态显示/隐藏逻辑
- 多个条件渲染增加维护成本
- 用户需要点击、切换才能看到所有选项

**改进建议：**
```diff
- 复杂的条件渲染和动态UI
+ 扁平化配置界面
+ 只显示核心配置项
+ 高级参数使用代码默认值
```

## 简化方案

### 方案A：激进简化（推荐新手用户）
**保留配置：**
1. 检索模式：向量 / 关键字 / 混合
2. TopK：返回结果数量
3. 相似度阈值：过滤低分结果

**删除配置：**
- BM25参数 → 固定为 k1=1.5, b=0.75
- 融合策略 → 固定为加权融合
- 权重配比 → 固定为 70%向量 + 30%关键字
- RRF参数 → 完全移除

**优点：**
- ✅ 配置项从8个降到3个
- ✅ 零学习成本，开箱即用
- ✅ UI极简，一目了然

**缺点：**
- ❌ 无法微调，灵活性降低

---

### 方案B：保守简化（推荐高级用户）
**保留配置：**
1. 检索模式：向量 / 关键字 / 混合
2. 权重配比：仅在混合模式显示（0-100%滑块）
3. TopK：返回结果数量
4. 相似度阈值：过滤低分结果

**删除配置：**
- BM25参数 → 固定为 k1=1.5, b=0.75
- 融合策略 → 固定为加权融合
- RRF参数 → 移除

**优点：**
- ✅ 配置项从8个降到4个
- ✅ 保留核心调参能力（权重配比）
- ✅ UI简洁，但有一定灵活性

**缺点：**
- ⚠️ 仍需理解"权重配比"概念

---

### 方案C：当前方案（不简化）
**保留所有配置：**
1. 检索模式 (3选1)
2. BM25参数 (k1, b)
3. 融合策略 (2选1)
4. 权重配比 / RRF参数
5. TopK
6. 相似度阈值

**优点：**
- ✅ 最大灵活性
- ✅ 适合算法研究人员

**缺点：**
- ❌ 复杂度高，学习成本高
- ❌ 维护成本高
- ❌ **违背KISS原则**

---

## 推荐决策

### 🎯 推荐：方案B（保守简化）

**理由：**
1. **平衡实用性和简洁性**
   - 保留最核心的调参能力（权重配比）
   - 隐藏专家参数（BM25/RRF）

2. **符合大多数用户需求**
   - 95%用户只需调整"向量vs关键字"的比例
   - 5%高级用户可通过修改代码调整BM25参数

3. **降低维护成本**
   - UI逻辑简化50%
   - 减少bug风险

4. **易于后期扩展**
   - 保留了代码实现
   - 未来可通过"高级模式"开关暴露更多参数
