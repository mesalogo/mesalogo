# 知识库文档处理按钮重构计划

## 需求背景
当前文档处理只有一个"转换"按钮，需要改造为支持三个独立步骤的下拉菜单形式，让用户能够灵活控制文档处理流程。

## 功能设计

### 1. 按钮结构
- **主按钮**："处理" - 一键执行完整流程（转换→分段→嵌入）
- **下拉菜单**：包含三个独立操作
  - 转换（Convert）：将原始文档转换为Markdown格式
  - 分段（Chunk）：将Markdown文档分割成合适的片段
  - 嵌入（Embed）：为文档片段生成向量嵌入

### 2. 状态管理

#### 文档处理状态流程
未处理 → 已转换 → 已分段 → 已嵌入

#### 状态字段
- `conversion_status`: 转换状态（not_converted/converting/converted/conversion_failed）
- `chunking_status`: 分段状态（not_chunked/chunking/chunked/chunking_failed）
- `embedding_status`: 嵌入状态（not_embedded/embedding/embedded/embedding_failed）

### 3. 操作逻辑

#### 3.1 按钮可用性

| 按钮 | 可用条件 | 说明 |
|------|---------|------|
| 主按钮"处理" | 始终可用 | 任何状态下都可以点击，执行完整流程 |
| 转换 | 始终可用 | 可随时重新转换文档 |
| 分段 | 已转换 | 必须先转换才能分段（未转换时禁用） |
| 嵌入 | 已分段 | 必须先分段才能嵌入（未分段时禁用） |

#### 3.2 数据依赖与清理规则
由于后续步骤依赖前面步骤的结果，重新执行会清理依赖数据：

| 操作 | 前置条件 | 清理范围 | 确认提示 |
|------|---------|----------|----------|
| 转换 | 无 | 清理分段+嵌入数据，重新生成转换 | "重新转换将清除分段数据和向量嵌入，是否继续？" |
| 分段 | 已转换 | 清理嵌入数据，重新生成分段 | "重新分段将清除向量嵌入，是否继续？" |
| 嵌入 | 已分段 | 重新生成嵌入 | "重新生成向量嵌入，是否继续？" |

#### 3.3 处理流程
1. **一键处理（主按钮）**
   - 始终从转换开始，完整执行三个步骤
   - 自动按顺序执行：转换→分段→嵌入
   - 如果文档已有部分处理，会清理并重新执行
   - 显示整体进度

2. **单步执行（下拉选项）**
   - 用户选择特定步骤
   - 系统检查前置条件（分段需已转换，嵌入需已分段）
   - 如果会清理后续数据，显示确认对话框
   - 确认后执行单个步骤
   - 完成后更新状态

### 4. UI表现

#### 4.1 按钮样式
```
主按钮：[处理 ▼] （始终可点击）
下拉菜单：
  ├─ 🔄 转换 (始终可用，显示状态：已完成✓/处理中.../失败✗)
  ├─ ✂️ 分段 (未转换时禁用，显示状态：已完成✓/处理中.../失败✗)
  └─ 📊 嵌入 (未分段时禁用，显示状态：已完成✓/处理中.../失败✗)
```

#### 4.2 状态指示
- **未处理**：灰色图标，普通文字
- **处理中**：加载动画，蓝色文字
- **已完成**：绿色勾号，绿色文字
- **失败**：红色叉号，红色文字

### 5. API接口需求

需要后端支持的API：
1. **分段API** `/api/knowledge/{id}/files/{file_path}/chunk`
   - POST: 执行分段操作
   - GET: 获取分段状态

2. **嵌入API** `/api/knowledge/{id}/files/{file_path}/embed`
   - POST: 执行嵌入操作
   - GET: 获取嵌入状态

3. **批量处理API** `/api/knowledge/{id}/files/{file_path}/process`
   - POST: 执行完整处理流程
   - 参数：可指定起始步骤

## 注意事项

1. **数据一致性**：确保清理操作的原子性，避免数据不一致
2. **并发控制**：防止同一文档同时执行多个处理操作
3. **错误处理**：明确的错误提示和恢复机制
4. **性能考虑**：大文件处理时的超时和分片策略
